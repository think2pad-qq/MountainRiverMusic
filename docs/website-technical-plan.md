# 普洁工作室网站建设方案

## 1. 目标定位

“普洁工作室”不应该只是一个能播放音乐的测试页，而应该成为专业音乐人的官方作品入口。网站的核心目标是：

- 展示原创音乐、配乐、器乐、口播或音乐故事等作品。
- 让访问者快速理解音乐人的风格、身份和创作方向。
- 沉淀平台之外的长期内容，包括创作笔记、作品背景、歌词、项目介绍。
- 提供清晰的合作、媒体、演出或商务联系入口。

第一阶段不追求复杂系统，重点是把网站做成稳定、清晰、可持续更新的静态作品站。

## 2. 推荐技术栈

### 核心方案

- 框架：Astro
- 语言：TypeScript
- 样式：CSS Modules 或普通 CSS；如果后续页面复杂，再引入 Tailwind CSS
- 内容管理：Markdown / MDX
- 部署：GitHub Pages
- 音频播放：HTML5 audio 起步，后续可升级为自定义播放器
- 图片资源：本地 `public/images/` 或外部图床/CDN
- 音频资源：小样可本地托管，正式或大量音频建议外部托管

### 为什么选择 Astro

Astro 适合内容型和作品型网站。音乐人官网的大部分页面是静态内容，例如首页、作品页、专辑页、关于页、Press Kit 和创作笔记。Astro 可以生成静态 HTML，页面加载快，部署简单，也利于搜索引擎收录。

相比纯 React/Vue SPA，Astro 更适合展示型网站；相比 Next.js，Astro 对当前需求更轻；相比 WordPress，Astro 不需要维护服务器和插件生态。

## 3. 不推荐的当前方案

### 暂不推荐 Next.js

Next.js 很适合复杂产品、会员系统、动态后台、支付和服务端渲染场景。但当前阶段主要是作品展示和内容沉淀，使用 Next.js 会增加工程复杂度。

### 暂不推荐 WordPress

WordPress 的后台编辑方便，但需要持续维护主题、插件、安全更新和服务器环境。对于一个音乐人官网，除非明确需要多人后台和非技术编辑流程，否则维护成本偏高。

### 暂不推荐纯 React/Vue 单页应用

纯 SPA 更像应用，不像作品站。它对首屏速度、SEO、社交分享卡片等方面不如静态生成站点自然。

## 4. 网站信息架构

建议第一版包含以下页面和内容模块。

### 首页

首页承担“第一印象”和“主推作品”的作用：

- 品牌名：普洁工作室
- 一句定位：例如“原创音乐、配乐与声音叙事”
- 主推作品播放器
- 代表作品入口
- 最新创作笔记入口
- 合作联系方式

### 作品库

作品库用于集中展示音乐作品。每个作品建议包含：

- 曲名
- 封面
- 年份
- 风格标签
- 简短介绍
- 音频播放器或外部平台链接
- 歌词、创作背景或制作说明

### 单曲/作品详情页

每首重点作品应有独立页面，便于分享和搜索收录。页面内容包括：

- 作品标题
- 音频播放器
- 封面图
- 创作说明
- 歌词或文本
- 制作人员信息
- 发布平台链接
- 相关推荐作品

### 关于页

关于页需要体现专业身份，而不是普通自我介绍。建议内容：

- 音乐人简介
- 风格方向
- 代表作品
- 可合作范围
- 联系方式

### Press Kit

如果你有演出、商务合作、媒体采访或项目合作需求，建议保留 Press Kit 页面：

- 短版简介
- 长版简介
- 艺术家照片
- Logo 或视觉标识
- 代表作品链接
- 联系方式

### 创作笔记

创作笔记是自建网站的长期价值：

- 作品幕后
- 编曲思路
- 乐器和设备记录
- 音乐观念
- 项目复盘

这些内容可以先用 Markdown 写，后续自然形成长期内容资产。

## 5. 内容数据结构

建议用 Markdown 或 MDX 维护作品，而不是把内容写死在 HTML 里。

示例结构：

```text
src/
  content/
    works/
      first-song.md
      mountain-demo.md
    notes/
      why-build-this-site.md
  components/
    AudioPlayer.astro
    WorkCard.astro
  layouts/
    BaseLayout.astro
  pages/
    index.astro
    works/
      index.astro
      [slug].astro
    notes/
      index.astro
      [slug].astro
    about.astro
    press-kit.astro
public/
  images/
  audio/
```

作品 Markdown 示例：

```md
---
title: "示例作品"
year: 2026
style: ["原创", "器乐", "氛围"]
cover: "/images/example-cover.jpg"
audio: "/audio/example.mp3"
externalLinks:
  bilibili: "https://example.com"
  netease: "https://example.com"
---

这里写作品背景、创作说明、歌词或制作札记。
```

## 6. 音频托管策略

GitHub Pages 可以托管静态文件，但不适合作为大量音频文件仓库。建议分阶段处理：

### 第一阶段

- 放少量压缩后的 MP3 小样到 `public/audio/`
- 每个音频尽量控制文件大小
- 保留外部平台链接

### 第二阶段

- 正式作品放到音乐平台、Bandcamp、SoundCloud、B站、网易云等
- 网站中嵌入或链接到这些平台
- 本站保留试听片段或精选版本

### 第三阶段

- 如果需要完整自主控制，再考虑对象存储和 CDN
- 可选方案包括 Cloudflare R2、阿里云 OSS、腾讯云 COS、七牛云等

原则：网站负责展示、叙事和转化；大文件分发交给更适合的音频平台或存储服务。

## 7. 视觉与体验方向

网站应体现专业音乐人的气质，不宜做成模板感很强的卡片堆叠页面。

建议风格：

- 安静、留白、清晰
- 首页第一屏突出“普洁工作室”和一首主推作品
- 色彩克制，但避免只有单一灰白
- 封面图、现场照、器材照或作品视觉要成为主要视觉资产
- 移动端播放器和作品列表必须易用

交互上先保持简单：

- 播放、暂停、进度条
- 作品筛选可以后置
- 波形、歌词同步、播放队列可以第二阶段再做

## 8. 分阶段实施计划

### 阶段一：从测试页升级为专业静态站

目标：保留 GitHub Pages 部署，建立 Astro 项目基础。

任务：

- 初始化 Astro + TypeScript
- 建立首页、作品库、关于页
- 建立基础布局和导航
- 建立作品 Markdown 数据结构
- 加入 2-3 个示例作品
- 使用 HTML5 audio 播放器
- 配置 GitHub Pages 自动部署

结果：网站从单个测试播放器变成可持续维护的音乐人官网。

### 阶段二：增强作品表达

目标：让每个作品有独立叙事空间。

任务：

- 增加作品详情页
- 增加歌词/创作背景/制作信息
- 增加 Press Kit 页面
- 增加创作笔记列表
- 加入 Open Graph 分享图和 SEO 元信息
- 优化移动端视觉

结果：每首作品都可以被单独分享、收藏和搜索。

### 阶段三：建立长期内容系统

目标：让网站成为长期资产。

任务：

- 接入独立域名
- 接入访问统计
- 接入联系表单或商务邮箱入口
- 接入外部音频平台或对象存储
- 设计更完整的播放器
- 根据作品数量增加分类、标签和搜索

结果：网站具备长期运营能力，可以承接听众、合作方和媒体访问。

## 9. 当前仓库建议改造路径

当前仓库只有：

```text
index.html
README.md
note.md
```

建议下一步不要继续在单个 `index.html` 中堆代码，而是执行 Astro 化改造：

1. 保留当前 `index.html` 作为历史参考。
2. 初始化 Astro 项目结构。
3. 将现有页面文案迁移到 `src/pages/index.astro`。
4. 新增 `src/content/works/`，让作品信息用 Markdown 管理。
5. 新增 GitHub Actions 部署流程。
6. 更新 README，让它反映真实技术栈和运行方式。

## 10. 第一版验收标准

第一版完成时，应满足：

- 访问 GitHub Pages 能看到完整首页。
- 首页有明确的专业音乐人定位。
- 至少有一个主推作品播放器。
- 至少有一个作品库列表。
- 至少有一个作品详情页。
- 有关于页和联系方式。
- 手机和桌面端都能正常阅读和播放。
- 新增作品只需要添加 Markdown 文件，不需要改页面代码。

## 11. 结论

推荐路线：

```text
Astro + TypeScript + Markdown/MDX + GitHub Pages + 外部音频托管
```

这个方案足够轻，适合从当前单页自然升级；同时又有清晰的扩展空间，后续可以承载作品库、创作笔记、Press Kit、独立域名、统计和更完整的播放器。
